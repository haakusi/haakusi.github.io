패러다임의 전환: 반응형에서 선제적 자동화로

전통적인 AI 시스템은 사용자의 명시적 요청에 반응하는 수동적 구조였습니다. 사용자가 질문을 입력하면 시스템이 응답을 생성하는 단순한 Request-Response 패턴이었죠. 하지만 수십, 수백개의 MCP Function Calling 도구가 유기적으로 연결된 시스템에서는 완전히 다른 패러다임이 가능합니다. 시스템이 내부 데이터를 지속적으로 수집하고 분석하여 사용자가 요청하기 전에 먼저 제안하고, 패턴을 학습하여 자동으로 실행하는 선제적 자동화 시스템으로의 진화입니다.

*다음 영상은 Agent와 각종 Tools가 어떻게 유기적으로 호출되고 동작하는지 만들어봤습니다. 주제는 팔란티어 시스템 입니다.

클릭하여 확대

왜 Function Calling이 대두 되었는가

MCP Function Calling은 AI가 단순히 텍스트 응답을 생성하는 것을 넘어 실제 시스템을 제어할 수 있게 합니다. 수십, 수백개의 도구가 유기적으로 연결되어 있을 때, AI는 데이터 수집, 패턴 분석, 이상 감지, 자동 조치까지 전체 사이클을 독립적으로 실행할 수 있습니다. 이는 단순한 추론 모델이 아닌 실행 가능한 자동화 시스템으로 변모시키는 기술이기 때문입니다.

데이터 수집: 핵심 IP 구축 전략

선제적 자동화의 첫 단계는 체계적인 데이터 수집입니다. 시스템은 사용자 입력, AI 응답, 내부 처리 과정, 시스템 메트릭을 실시간으로 수집합니다. 중요한 점은 사용자 목적을 자동으로 분류한다는 것입니다. 문제 해결 목적인지, 인사이트 리포트 생성인지, 데이터 분석인지, 컴플라이언스 점검인지, 정책 최적화인지를 자동으로 라벨링하여 고객사별 사용 패턴을 세그멘테이션합니다. 의도 실패율 데이터를 추적하여 목표 달성 여부, 재요청 횟수, 응답 만족도를 분석하고 자동화가 필요한 기능을 식별합니다.

수집 데이터 구조:
├── 기본 실행 로그
│   ├── User Input (사용자 질문/요청)
│   ├── AI Response (AI 응답 결과)
│   ├── Internal Thinking/React (처리 과정)
│   └── System Metrics (CPU, 메모리, 디스크)
│
├── 사용자 목적 분류 (자동 라벨링)
│   ├── 문제 해결 목적 (35%)
│   ├── 인사이트 리포트 생성 (28%)
│   ├── 데이터 분석 목적 (22%)
│   ├── 컴플라이언스 점검 (10%)
│   └── 정책 최적화 목적 (5%)
│
└── 의도 실패율 데이터
    ├── 목표 달성 여부 (YES/NO)
    ├── 재요청/수정/반복 횟수
    ├── 응답 만족도 추적
    └── 실패 원인 분석

데이터 분석: 인사이트 도출 전략

수집된 데이터에서 실행 가능한 인사이트를 도출합니다. 도구 사용 빈도를 분석하여 query-identities가 850회, analyze-intelligence-feed가 720회 호출되었다면 가장 자주 사용되는 기능에 대한 자동화 스크립트를 우선 제공합니다. 카테고리별 요청 분포를 파악하여 Identity Management가 40%, Intelligence Analysis가 20%라면 우선순위 기반 기능 개선 로드맵을 수립합니다. 사용자 목적 자동 분류를 통해 "이 고객사는 인사이트 리포트 자동화 중심", "이 고객사는 실시간 인텔리전스 모니터링 중심"으로 세그멘테이션하여 맞춤형 서비스를 제공합니다.

선제적 자동화 시나리오 1: 인기 기능 자동화

특정 도구와 인자 조합이 하루 10회 이상 반복되면 시스템이 자동으로 감지합니다. 예를 들어 query-identities + {"active": true}가 하루 15회 호출되면 "이 작업을 매일 자동 실행하시겠습니까?" 배너를 표시합니다. 사용자가 승인하면 스케줄을 자동 생성하고, 매일 오전 9시에 query-identities를 실행하여 활성 엔티티 목록 리포트를 이메일이나 슬랙으로 발송합니다. 이는 사용자가 반복 작업을 인식하기도 전에 시스템이 먼저 제안하는 선제적 접근입니다.

선제적 자동화 시나리오 2: 패턴 기반 자동 정책 생성

시스템은 평일 09:00-18:00 정상 행동 패턴을 학습하고, 이후 접근 시도의 대부분이 실패한다는 것을 감지합니다. 그러면 "18시 이후 접근은 관리자 승인만 허용할까요?"라는 정책 최적화를 자동으로 추천하고, 사용자 승인 시 update-authorization-policy 도구를 자동 호출하여 time_restriction="after_18:00", require_admin_approval=True 정책을 실제로 적용합니다. AI가 단순히 제안만 하는 것이 아니라 Function Calling을 통해 실제 시스템을 변경합니다.

선제적 자동화 시나리오 3: 이상행동 보안 알림

평소 패턴 대비 특정 시간대의 접근 이벤트가 급증하거나, 실패/거부 비율이 급증하거나, 야간/휴일 특정 엔티티 접근 빈도가 갑자기 증가하면 시스템이 자동으로 대응합니다. analyze-intelligence-feed로 이상 패턴을 상세 조회하고, query-identity-registry로 엔티티 정보를 확인하며, update-authorization-policy로 임시 접근을 자동 제한하고, control-access-point로 특정 접근 지점을 임시 잠금합니다. 동시에 보안 담당자에게 즉시 알림과 상세 리포트를 자동 생성합니다. 사용자 개입 없이 탐지 → 분석 → 조치가 자동으로 실행됩니다.

선제적 자동화 시나리오 4: 오류 패턴 기반 선제적 조치

auth_error가 지난 1시간 동안 10회 이상 발생하고 특정 사용자에게 반복되면, 시스템은 사용자에게 "세션 만료가 가까워집니다. 재로그인 하시겠습니까?" 알림을 표시하고, 관리자에게는 슬랙/이메일로 "이 IP 대역에서 인증 오류 급증" 리포트를 자동 발송하며, 시스템 레벨에서는 자동 재시도 정책 적용과 rate limit 조정을 제안합니다. 오류가 발생한 후 대응이 아닌 오류가 발생하기 전 예방입니다.

선제적 자동화 시나리오 5: 성능 최적화 자동 제안

평균 응답 시간이 1시간 이동평균 기준 임계값 이상으로 증가하거나 특정 도구에서 latency가 급증하면, 대시보드에 "성능 이슈 감지" 배너를 표시하고 캐싱 후보를 자동 추천합니다. "query-identities 최근 1시간 120회 호출 → 캐시 TTL 5분 적용 추천"과 같은 구체적 제안을 하고, 관리 UI에서 "캐시 적용" 버튼 하나로 설정을 즉시 변경할 수 있게 합니다.

데이터 → 인사이트 → 자동화 플로우

구체적인 동작 흐름을 살펴봅시다. 백그라운드 모니터링이 매 시간마다 데이터 스트림을 분석하다가 "비정상 접근 시도 40회 (평소 10회 대비 300% 증가)"를 감지합니다. 임계값 200%를 초과하여 자동 트리거가 발생하고, LLM이 수집된 인텔리전스 데이터를 컨텍스트로 받아 원인을 분석합니다. "신규 엔티티 프로비저닝 누락" 또는 "권한 정책 설정 오류"로 진단하고, 대시보드에 "자동 엔티티 프로비저닝" 또는 "권한 정책 재검토" 버튼을 표시합니다. 사용자가 클릭하면 provision-identity, update-authorization-policy 등 MCP 도구가 자동으로 호출되어 문제를 해결합니다. 전체 과정이 데이터 기반으로 자동화됩니다.

서비스 진화 단계

Phase 1은 API Wrapper 기반 인터페이스로 UI와 MCP 도구를 Claude/GPT/Gemini에 연결하여 기본 기능을 제공합니다. Phase 2는 사용량 분석 단계로 로그 수집, 통계 시각화, 함수 호출 분석을 수행합니다. Phase 3은 AI 지원 자동화로 자동 리포트, 주기 실행, 자동 알림을 구현합니다. Phase 4는 행동 기반 ML로 사용 패턴 모델링, 예측 기반 자동화, 추천 엔진을 도입합니다. Phase 5는 자율 보안 자동화로 이상 행동 탐지, 비정상 접근 자동 대응, 정책 자동 최적화를 실현합니다.

핵심 차별화 포인트

첫째, 데이터 소유권입니다. 외부 시스템 API로부터 직접적인 접점 없이 모든 데이터를 우리가 소유하며 이것이 핵심 IP가 됩니다. 둘째, Function Calling 기반 실행입니다. AI가 분석만 하는 것이 아니라 MCP Tools를 통해 실제로 조치를 실행하고 시스템을 직접 제어하여 사용자 개입을 최소화합니다. 셋째, 선제적 자동화입니다. 문제가 발생한 후 대응이 아닌 문제 발생 전 예방, 패턴 기반 예측 분석, 자동 트리거 및 실행을 구현합니다. 넷째, 인사이트 중심입니다. 단순 추론이 아닌 데이터 기반 인사이트를 제공하고, 고객사별 세그멘테이션과 맞춤형 서비스를 제공합니다.

결론: GPU 없이 구현하는 차세대 AI 자동화

MCP Function Calling을 기반으로 한 선제적 자동화 시스템은 GPU 투자 없이도 강력한 AI 제품을 만들 수 있음을 보여줍니다. 핵심은 추론 모델이 아닌 데이터 수집과 인사이트 도출입니다. 수십, 수백개의 도구가 유기적으로 연결된 시스템에서 데이터 수집으로 핵심 IP를 구축하고, 데이터 분석으로 인사이트를 도출하며, Function Calling으로 실제 시스템을 제어하여 선제적 자동화를 구현하는 완전한 AI 기반 엔터프라이즈 인텔리전스 자동화 플랫폼이 가능합니다. 이는 반응형 AI에서 선제적 AI로의 패러다임 전환이며, 데이터 기반 자동화의 미래를 보여주는 구체적 사례입니다.

Paradigm Shift: From Reactive to Proactive Automation

Traditional AI systems were passive structures that reacted to explicit user requests. It was a simple Request-Response pattern where the system generated responses when users entered questions. However, in a system where dozens, hundreds of MCP Function Calling tools are organically connected, a completely different paradigm becomes possible. The evolution to a proactive automation system where the system continuously collects and analyzes internal data to suggest before users request, learns patterns to execute automatically.

Why Function Calling Has Emerged

MCP Function Calling enables AI to go beyond generating text responses to actually controlling real systems. When dozens, hundreds of tools are organically connected, AI can independently execute the entire cycle from data collection, pattern analysis, anomaly detection to automatic actions. This is one of the technologies that transforms it from a simple inference model into an executable automation system.

Data Collection: Core IP Building Strategy

The first step of proactive automation is systematic data collection. The system collects user input, AI responses, internal processing, and system metrics in real-time. Importantly, it automatically classifies user purposes. Whether it's problem-solving, insight report generation, data analysis, compliance checking, or policy optimization is automatically labeled to segment usage patterns by customer. Intent failure rate data tracks goal achievement, retry counts, response satisfaction to identify features needing automation.

Data Collection Structure:
├── Basic Execution Logs
│   ├── User Input (questions/requests)
│   ├── AI Response (response results)
│   ├── Internal Thinking/React (processing)
│   └── System Metrics (CPU, memory, disk)
│
├── User Purpose Classification (auto-labeling)
│   ├── Problem Solving (35%)
│   ├── Insight Report Generation (28%)
│   ├── Data Analysis (22%)
│   ├── Compliance Checking (10%)
│   └── Policy Optimization (5%)
│
└── Intent Failure Rate Data
    ├── Goal Achievement (YES/NO)
    ├── Retry/Modification Count
    ├── Response Satisfaction Tracking
    └── Failure Cause Analysis

Data Analysis: Insight Derivation Strategy

Actionable insights are derived from collected data. By analyzing tool usage frequency, if query-identities was called 850 times and analyze-intelligence-feed 720 times, automation scripts are prioritized for the most frequently used features. Understanding category-wise request distribution, if Identity Management is 40% and Intelligence Analysis is 20%, feature improvement roadmaps are established based on priority. Through automatic user purpose classification, customers are segmented as "this customer focuses on insight report automation" or "this customer focuses on real-time intelligence monitoring" to provide customized services.

Proactive Automation Scenario 1: Popular Feature Automation

When a specific tool and argument combination repeats more than 10 times per day, the system automatically detects it. For example, if query-identities + {"active": true} is called 15 times daily, a banner displays "Would you like to automate this task daily?" Upon user approval, a schedule is automatically created to execute query-identities every morning at 9 AM and send active entity list reports via email or Slack. This is a proactive approach where the system suggests before users even recognize the repetitive task.

Proactive Automation Scenario 2: Pattern-Based Auto Policy Generation

The system learns normal behavior patterns during weekdays 09:00-18:00 and detects that most access attempts afterward fail. It then automatically recommends policy optimization: "Allow access after 6 PM only with admin approval?" Upon user approval, it automatically calls the update-authorization-policy tool to apply the policy with time_restriction="after_18:00" and require_admin_approval=True. AI doesn't just suggest—it actually changes the real system through Function Calling.

Proactive Automation Scenario 3: Anomaly Behavior Security Alerts

When access events surge during specific times compared to usual patterns, failure/rejection rates spike, or specific entity access frequency suddenly increases at night/holidays, the system automatically responds. It queries anomaly patterns in detail with analyze-intelligence-feed, verifies entity information with query-identity-registry, automatically restricts temporary access with update-authorization-policy, and temporarily locks specific access points with control-access-point. Simultaneously, it automatically generates alerts and detailed reports for security personnel. Detection → Analysis → Action executes automatically without user intervention.

Proactive Automation Scenario 4: Error Pattern-Based Proactive Actions

When auth_error occurs more than 10 times in the past hour and repeatedly for specific users, the system displays "Session expiring soon. Would you like to re-login?" to users, automatically sends Slack/email reports to admins about "authentication error surge from this IP range," and at the system level suggests applying automatic retry policies and rate limit adjustments. This is prevention before errors occur, not response after they happen.

Proactive Automation Scenario 5: Performance Optimization Auto Suggestions

When average response time increases above threshold based on 1-hour moving average or latency spikes for specific tools, the dashboard displays a "Performance Issue Detected" banner and automatically recommends caching candidates. It provides specific suggestions like "query-identities called 120 times in the last hour → recommend applying 5-minute cache TTL" and enables immediate configuration changes with a single "Apply Cache" button in the admin UI.

Data → Insight → Automation Flow

Let's examine the specific operation flow. Background monitoring analyzes data streams every hour and detects "40 abnormal access attempts (300% increase from usual 10)." Exceeding the 200% threshold triggers automatic activation, and the LLM receives collected intelligence data as context to analyze the cause. It diagnoses "new entity provisioning missing" or "permission policy configuration error" and displays "Auto Entity Provisioning" or "Review Permission Policy" buttons on the dashboard. When users click, MCP tools like provision-identity and update-authorization-policy are automatically called to resolve the issue. The entire process is data-driven automation.

Service Evolution Stages

Phase 1 is an API Wrapper-based interface connecting UI and MCP tools to Claude/GPT/Gemini providing basic functionality. Phase 2 is the usage analytics stage performing log collection, statistics visualization, and function call analysis. Phase 3 is AI-assisted automation implementing automatic reports, periodic execution, and automatic notifications. Phase 4 is behavioral ML introducing usage pattern modeling, prediction-based automation, and recommendation engines. Phase 5 is autonomous security automation realizing anomaly behavior detection, abnormal access automatic response, and policy auto-optimization.

Key Differentiators

First, data ownership. We own all data without direct touchpoints from external system APIs, making this our core IP. Second, Function Calling-based execution. AI doesn't just analyze—it executes actions through MCP Tools, directly controls systems, and minimizes user intervention. Third, proactive automation. Prevention before problems occur, not response after, implementing pattern-based predictive analysis and automatic triggering and execution. Fourth, insight-centric. Providing data-based insights not simple inference, customer segmentation, and customized services.

Conclusion: Next-Gen AI Automation Without GPUs

The proactive automation system based on MCP Function Calling demonstrates that powerful AI products can be created without GPU investment. The key is not inference models but data collection and insight derivation. In a system where dozens, hundreds of tools are organically connected, building core IP through data collection, deriving insights through data analysis, controlling real systems through Function Calling to implement proactive automation—a complete AI-based enterprise intelligence automation platform is possible. This is a paradigm shift from reactive AI to proactive AI, a concrete example showing the future of data-driven automation.
