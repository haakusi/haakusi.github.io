00 / POSITIONING

I turn boundaries into products.

My work sits where hardware, enterprise software, and AI meet. I do not position myself as a model researcher who happens to ship code, or as a full-stack engineer who happens to use LLMs. My recurring job is to make difficult boundaries usable: device to server, server to web, API to developer, agent to real-world action, and fast generation to reliable release.

제 일은 하드웨어, 엔터프라이즈 소프트웨어와 AI가 만나는 곳에 있습니다. 코드를 배포하는 모델 연구자도, LLM을 사용하는 일반 풀스택 개발자도 아닙니다. 장치와 서버, 서버와 웹, API와 개발자, 에이전트와 실제 행동, 빠른 생성과 신뢰 가능한 릴리스 사이의 어려운 경계를 사용 가능한 제품으로 만드는 일을 반복해 왔습니다.

CASE 01

A developer portal that behaves like a product system

AWARD · PRODUCTIZING
Context

Product APIs, SDK behavior, execution environments, and support knowledge were fragmented. A developer's first successful call depended too much on local setup and expert help.

제품 API, SDK 동작, 실행 환경과 지원 지식이 분절돼 있었습니다. 개발자가 첫 호출에 성공하려면 로컬 설정과 전문가 지원에 과도하게 의존해야 했습니다.

My decision

Build six connected surfaces—not another documentation site. Docs, an API source of truth, a device emulator, MCP tools, a Linux/AWS sandbox, and evaluation gates share one operation model.

문서 사이트를 하나 더 만드는 대신 여섯 개의 제품 표면을 연결했습니다. 문서, API 단일 기준, 장치 에뮬레이터, MCP 도구, Linux/AWS 샌드박스와 평가 게이트가 같은 operation model을 참조합니다.

Natural-language verification

Through MCP, developers can recall complex user, door, authentication, and event scenario history in natural language, replay it inside the Linux/AWS sandbox, and verify it immediately. The same gateway is designed to extend from the emulator to physical devices.

MCP로 사용자·도어·인증·이벤트의 복합 시나리오 이력을 자연어로 호출하고 Linux/AWS 샌드박스에서 재실행해 즉시 검증합니다. 같은 게이트웨이를 에뮬레이터에서 실제 장치까지 확장하도록 설계했습니다.

Current state

Built solo in eight weeks with an AI-native workflow, recognized with a company-wide excellence award, and now being hardened toward a September 2026 release target.

AI-native workflow로 1인 8주 동안 구축해 전사 우수상을 받았고, 현재 2026년 9월 목표 릴리스를 위해 보안·운영·지원 기준을 제품 수준으로 강화하고 있습니다.

CASE 02

AI-native modernization with meaning-preservation gates

IN PROGRESS
Problem

A large Angular frontend embedded in an iframe had evolved into a mixed Angular/React architecture with tangled routing, back navigation, and state boundaries.

iframe 안의 대규모 Angular 프론트엔드가 Angular/React 혼합 구조로 발전하면서 routing, 뒤로가기와 상태 경계가 얽힌 기술 부채가 누적됐습니다.

Approach

I lead the frontend team with an AI-native migration harness that checks backend payloads and responses, API contracts, React behavior, useEffect, DOM, UI, CSS assertions, and E2E flows as separate release gates.

프론트팀을 리딩하며 backend payload/response, API contract, React 동작, useEffect, DOM, UI, CSS assertion과 E2E를 독립 릴리스 게이트로 다루는 AI-native migration harness를 설계했습니다.

PRINCIPLE

Generated code is not progress until equivalent behavior is proven.

02 / OPERATING MODEL

Verification-first AI-native engineering

01

Define the real boundary

Start from the device, user, developer, or release failure—not from the model or framework.

02

Establish one source of truth

Make docs, APIs, tools, sandboxes, and tests share the same meaning before optimizing speed.

03

Design failure gates

Use unit, contract, DOM, UI, CSS, E2E, and evaluation layers so each type of wrong result fails visibly.

04

Turn a method into team leverage

Convert individual techniques into reusable harnesses, examples, and operating rules that other teams can adopt.
